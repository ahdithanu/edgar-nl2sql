The pipeline

Five steps per request. The loop between execute and generate is the interesting part.

01
Embed & retrieve
The question is embedded and matched against schema chunks and a business-term glossary in pgvector — so the model sees the right tables and definitions before it writes anything.
02
Generate with schema in prompt
Retrieved context is assembled into the prompt and the model emits a single SQL statement — no free-roaming table access, just the columns it was shown.
03
Validate read-only & execute
The statement is parsed and rejected unless it's a single SELECT. Only then does it run against Postgres, inside a READ ONLY transaction with a statement timeout.
04
Feed the error back & retry — capped at 3
If execution errors or returns nothing, the exact failure is fed back to generate for another attempt. Hard cap of 3 attempts bounds latency and cost; past that it fails loud, not silent.
05
Synthesize the answer
Result rows are handed back to the model to produce a plain-English answer alongside the raw table — so a reader gets both the takeaway and the receipts.

Engineering discipline

The model is the easy part. These are the guardrails that make it shippable.

eval harness
17 golden questions
A fixed set of question→expected-result pairs run on every change. Accuracy is a number, not a vibe.
ci gate
Fails the build below the floor
The eval runs in CI and blocks merge if accuracy drops under the threshold. Regressions can't ship.
observability
JSON logs, every attempt traced
Retrieval hits, generated SQL, validation, and each retry are logged structured with a request id — a failure is reproducible.
deploy
staging → prod
Changes land in staging behind the same eval gate before promotion. Prod is boring on purpose.
safety
SELECT-only enforcement
Parsed at the app layer and backed by a READ ONLY transaction. Two independent locks, not one prompt instruction.
cost
3-attempt cap
Retries are bounded so a pathological question can't burn unbounded tokens or wall-clock time.

Deliberately not built

Scope is a design tool. What's left out is a decision, not an oversight.

✕
Multi-cloud / AWS build-out. One managed Postgres and one container host. Portability isn't a requirement this project has.
✕
Separate vector service. pgvector lives in the same Postgres. No second datastore to operate for 25 companies of filings.
✕
Premature scaling infra. No queues, no sharding, no autoscaling groups. The load doesn't exist yet, so neither does the machinery for it.
✕
Multi-origin routing. Single origin, single region. Edge distribution buys nothing for an internal-style tool with a handful of users.
